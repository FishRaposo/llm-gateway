## Data Flow - Request Processing

```mermaid
sequenceDiagram
    participant App as Application
    participant Proxy as Gateway Proxy
    participant Auth as Auth
    participant Policy as Policy
    participant Budget as Budget
    participant Cache as Cache
    participant Router as Router
    participant Provider as LLM Provider
    participant Log as Audit Log

    App->>Proxy: POST /v1/chat/completions
    Proxy->>Auth: Validate API key
    Auth-->>Proxy: Valid / Invalid

    alt Invalid
        Proxy-->>App: 401 Unauthorized
    else Valid
        Proxy->>Policy: Check content policy
        Policy-->>Proxy: Allowed / Blocked

        alt Blocked
            Proxy-->>App: 403 Policy violation
        else Allowed
            Proxy->>Budget: Check remaining budget
            Budget-->>Proxy: Within limit / Exceeded

            alt Exceeded
                Proxy-->>App: 429 Budget exceeded
            else Within limit
                Proxy->>Cache: Check cache
                Cache-->>Proxy: Hit / Miss

                alt Cache Hit
                    Proxy->>Budget: Update spend estimate
                    Proxy-->>App: Cached response
                else Cache Miss
                    Proxy->>Router: Route to provider
                    Router->>Provider: Forward request
                    Provider-->>Router: Response
                    Router-->>Proxy: Response
                    Proxy->>Cache: Store response
                    Proxy->>Budget: Update spend
                    Proxy->>Log: Log request/response
                    Proxy-->>App: Response
                end
            end
        end
    end
```

## Key Architectural Decisions

| Decision | Rationale |
|----------|-----------|
| **Middleware Chain** | Each concern isolated, testable, composable; order matters |
| **OpenAI-compatible API** | Zero migration cost; existing SDKs work by changing base URL |
| **Redis for Hot Data** | Sub-millisecond latency for caching and rate limiting |
| **SQLite for Audit** | Append-heavy, query-light; zero operational overhead |
| **Circuit Breaker** | Fail fast when provider is unhealthy; improves reliability |
| **Provider Adapters** | Abstract provider differences; easy to add new providers |

## Performance Characteristics

| Metric | Target |
|--------|--------|
| p99 latency overhead | <10ms (cached) |
| p99 latency overhead | <50ms (uncached) |
| Cache hit rate | >60% for repeated queries |
| Rate limit precision | Sliding window, 1-second granularity |
| Provider failover | <100ms detection |
| Concurrent requests | 1000+ per instance |

## Security Model

| Layer | Protection |
|-------|------------|
| Auth | API key validation with bcrypt hashes |
| Policy | Regex + keyword content filtering |
| Rate Limit | Per-key sliding windows |
| Budget | Per-key spend tracking |
| Audit | Full request/response logging |
